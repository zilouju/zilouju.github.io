# UTF-8 乱码问题记录

## 问题现象

发中文动态后，页面或数据文件中出现类似下面的乱码：

```text
你好 -> ä½ å¥½
你好 -> Ã¤Â½Â Ã¥Â¥Â½
```

这类内容不是中文本身损坏，而是 UTF-8 字节被按 Latin-1/单字节字符串错误解释后的结果。第二种形式通常是同一份乱码又被再次错误编码或解码后形成的双重乱码。

## 根本原因

GitHub Contents API 返回文件内容时，`content` 字段是 Base64。旧实现曾直接使用 `atob(...)` 后把结果当普通 JavaScript 字符串传给 `JSON.parse`。

`atob` 返回的是按字节拼出的“二进制字符串”。中文在 UTF-8 中由多个字节组成，如果没有用 `TextDecoder('utf-8')` 把这些字节还原成真正的 UTF-8 文本，浏览器就会把每个字节当作单独字符处理，导致：

```text
UTF-8 字节 E4 BD A0 E5 A5 BD
错误当作单字节字符读取
显示为 ä½ å¥½
```

更严重的是，页面保存时又把已经乱码的内容写回 `data/posts-recent.json`，因此乱码从“显示问题”变成了“持久化数据问题”。

## 关键修复点

1. GitHub API 读写统一显式使用 UTF-8。

读取 Base64 时，先 `atob` 得到字节，再用 `TextDecoder('utf-8')` 解码。写入时，先用 `TextEncoder` 把 JSON 字符串编码为 UTF-8 字节，再 Base64。

2. 增加历史乱码修复。

`fixMojibake` 会识别类似 `ä½ å¥½` 的 Latin-1 误读内容，并按 UTF-8 字节还原；同时支持多轮处理，用于修复双重乱码。

3. 保存前统一规范化数据。

`saveAll` 在写入 `posts-recent.json` 和 `posts-archive.json` 前会调用 `normalizePosts`，避免把已损坏的内容再次写回。

4. 发布成功后的回显使用规范化后的状态。

`createPost` 保存后不再使用保存前的 `all` 数组渲染，而是从更新后的 `state` 中通过 `getAllPosts()` 获取内容，确保页面回显和最终写入的数据一致。

## 验证方式

本地模拟发帖后，应同时确认三个位置都是正常中文：

```text
发帖输入: 你好，本地发帖测试
发布后回显: 你好，本地发帖测试
写入 JSON 首条: 你好，本地发帖测试
```

同时检查 `data/` 目录内不再出现 `ä½`、`Ã¤` 等历史乱码片段。
